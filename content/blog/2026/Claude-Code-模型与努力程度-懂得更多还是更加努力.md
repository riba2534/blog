![模型与努力程度：两个设置的概念示意](https://img.riba2534.cn/images/2026/07/cc-model-effort-01-hero.png)

Claude Code 给了你两个看起来都能"让回答更好"的设置：**模型**，和**努力程度**（effort level）。但它们到底分别对输出做了什么？你又该怎么判断，是该换一个模型，还是只调一下努力程度？

人很容易想当然：选一个更大的模型（比如 Fable）会比 Sonnet 给出更聪明的输出，而更高的努力程度无非意味着 Claude 在回答前多想一会儿。

第一个假设是对的。按照业界标准的基准测试，我们最大的模型确实能力更强。

但努力程度的意义远不止"思考时间"。**努力程度控制的是 Claude 为你的请求所做的全部工作量。**这包括它思考多久，但也包括：

- 它读多少文件；
- 它做多少验证；以及
- 在一个多步任务里，它在回头找你之前能推进多远。

在更高的努力程度下，Claude 会在回到你这里之前做更多这类动作（读文件、跑测试、反复核对）。而在更低的努力程度下，它宁愿找你要更多上下文，也不愿自己花 token 去把某件事搞清楚。

## 模型选择是如何工作的

要理解模型这个设置到底控制了什么，不妨从头说起——从你按下回车的那一刻开始。

Claude Code 会把你的消息连同系统提示词、工具定义、你的 CLAUDE.md、对话历史，以及上下文中的所有文件拼装到一起。所有这些，作为一个请求发送给 API。

![Claude Code 收集的所有东西都被打包进一个 API 请求](https://img.riba2534.cn/images/2026/07/cc-model-effort-02-api-request.png)

在服务器上，文本在任何模型接触它之前，会先被分词。

不过，模型从来不会把这些当成纯文本来看到。服务器上发生的第一件事是分词（tokenization）：文本被切成片段，每个片段被映射到一个整数，这个整数来自模型训练时所用的固定词表。`const` 可能映射到 1978，`await` 可能映射到 4293。从这一步起，你的提示词就是一个整数数组。

![分词器把你的文本切成片段，并把每个片段映射到固定词表中的一个整数。顶行的每个块都变成它在底行的 token ID；所示 ID 仅为示意](https://img.riba2534.cn/images/2026/07/cc-model-effort-03-tokenizer.png)

模型的任务就是拿到那个数组，预测下一个 token 是什么。它通过对词表中的每个 token 计算一个概率，然后从概率最高的那些里挑一个来完成这件事。在 `const x = await` 之后，一个训练良好的模型会给 `fetch` 很高的概率（非常可能），而给 `banana` 接近零的概率（完全不可能）。

![模型的预测是对其词表中每个 token 的一个概率。排在前面的那个猜测，与一个无关词之间的差距是巨大的](https://img.riba2534.cn/images/2026/07/cc-model-effort-04-probability.png)

把你的输入 token 变成那些概率的，是**权重**（也叫参数）：数十亿个数字，组织成一个个大矩阵。为了预测一个 token，模型把你的输入送过这些矩阵（一长串矩阵乘法），然后在末端读出概率。模型"知道"的一切，都活在权重里。

每个模型的权重是在训练时设定的，等到你发送请求的时候，它们是**只读**的。你的提示词、你的 CLAUDE.md，或者你的上下文里的任何东西，都不会改变它们。如果你碰到过"推理"（inference）这个词，它的全部含义就是这个：在训练完成之后使用模型，权重是固定的。

![你的提示词进去，概率出来。中间的权重不会改变](https://img.riba2534.cn/images/2026/07/cc-model-effort-05-weights.png)

Claude 关于 TypeScript、各种流行框架，或者任何其他通用编程知识的一切，都是在训练时被编码进那些权重里的。

你的提示词和上下文仍然可以引导（steer）预测。把你真实的代码放到 Claude 面前，就是一种引导，而且效果非常好。但这并不会给权重本身增加任何东西。

如果某个库在模型训练时还不存在，它就不在权重里。你可以把文档放进上下文，Claude 会去用它，但那是引导，不是教学。Claude 的响应只在那一个请求里受到影响，而底层的模型什么也没有记住。

当 Claude 自信地调用一个根本不存在的 API 时（一种幻觉），那是权重根据训练中见过的模式，生成了一段看起来说得过去的 token 序列，而不是一次失败的查询。

那么，切换模型到底做了什么？**它换了一组冻结的权重，来处理你的请求。**

模型不会一次性生成整个答案。它预测一个 token，把它追加到序列里，然后重新跑一遍完整的计算来得到下一个。一个 200 个 token 的回答，就是 200 次单独地穿过权重。你大部分的等待时间（以及你的输出成本）都花在这个循环上。

![序列每一步恰好增长一个 token。模型每次都重新读取整个数组，以预测下一个是什么](https://img.riba2534.cn/images/2026/07/cc-model-effort-06-sequence.png)

模型这个设置决定了由哪些权重来处理你的请求，也决定了每个输出 token 的价格。

它不决定的，是要生成多少个 token。同样的提示词，这个数字可以差别很大，取决于 Claude 决定做多少工作。

而这恰恰就是努力程度所控制的。

## 努力程度是如何工作的

当 Claude Code 在处理一个任务时，它生成的 token 分成几类：

- **思考**（Thinking）：你在动作之前和之间看到流式输出的那段推理。
- **工具调用**（Tool calls）：一些结构化的块，指明一个工具（比如 Read 或 Edit）及其参数，然后 Claude Code 会去解析并执行它们。
- **给你的文本**（Text to you）：计划、进度更新，以及最后的总结。

这些都是同一个循环里产生的普通输出 token，按同样的费率计费。比如，思考 token 和其他输出 token 的生成方式完全一样，并会在这一轮的剩余时间里留在上下文里。

等到 Claude 开始写代码的时候，它先前的推理已经成了输入的一部分，就像它读过的一个文件一样。

![Claude 的所有输出都是 token。思考、工具调用和给你的文本，都由同一个循环生成](https://img.riba2534.cn/images/2026/07/cc-model-effort-07-output-tokens.png)

那么，努力程度又是怎么改变这一切的？努力程度作为请求的一部分，和你的提示词一起被发送给模型。模型经过训练，懂得在每个努力程度下该如何表现，而这种学到的行为，被烘焙进了那些冻结的权重里。

当你的请求到达时，努力程度只是模型要响应的又一个输入，就像它响应你的提示词文本一样。**它设定了 Claude 在认为任务完成之前，需要做到多彻底、多确定。**这在每一轮里都会被权衡，而更高的置信度需要更多 token 才能达到。

![同一个提示词，两个努力程度。高努力程度的路径生成的 token 大约是低者的 7 倍，以到达一个置信度更高的答案](https://img.riba2534.cn/images/2026/07/cc-model-effort-08-two-effort.png)

在更高的努力程度下，Claude 往往会先制定一个计划，而努力程度会影响这个计划的深度和广度。但计划并不是固定死的。随着 Claude 从它的动作里拿回结果，它会更新对自己进展程度、以及对累积结果确定程度的判断。

当一个三假设调试计划的第一步就找到了 bug，"调查假设 2 和 3" 可能就不再必要了。Claude 通常会把这一点明说出来（比如"第一项检查就找到了，所以剩下的检查不需要了"），然后直接跳过。你在 Claude Code 里会看到这种现象——任务列表会在运行中途被修改。

更高的努力程度，确实会让 Claude 更倾向于反复核对，比如验证它找到的答案，或者仍然去研究那些它本可以跳过的假设。不过，它通常不会仅仅因为努力程度被调高，就在一个简单任务上人为地膨胀用量。"过度思考"是我们团队在模型训练时专门盯防的东西，因为它会削弱效果。

## 选择努力程度

对大多数任务来说，用模型的默认努力程度就好。默认值是这样一档：Claude 会把它的 token 用量，调整到大多数人愿意在一个任务上花费的程度。

把努力程度当作一个手动的覆盖开关，控制 Claude 干得多起劲、干多久。当基于你的领域或你做的事，你对彻底性或速度有强烈偏好时，再刻意去动它；并且把它当作一种总体偏好，而不是逐任务的决定。

随着 Opus 4.8 发布，有一个实用的提示：在我们的测试里，Opus 4.8 的默认努力程度设置，在同一个任务上，用大约同样数量的 token，产出了比 Opus 4.7 默认努力程度更好的结果。

## Claude 出错时该改什么

当 Claude 把某件事搞错了，你的第一反应不该是去改设置，而应该是去看看你给它的上下文。你的提示词是不是太含糊？Claude 是不是连上了对的工具？它有没有对的技能？

如果你在一个本不需要高努力程度的任务上去调高它，问题的修正通常在上游：在你的上下文、你的 CLAUDE.md，或者任务的界定方式里。

但假设你已经给了清晰的上下文，Claude 还是搞错了。这时候该问自己的问题是：**它是努力得不够，还是懂得不够？**

![出错时，先判断是 Claude 懂得不够，还是努力得不够——这决定了该动模型还是努力程度](https://img.riba2534.cn/images/2026/07/cc-model-effort-09-decision.png)

### 模型：问题太难了

当问题确实难的时候——比如微妙的 bug、陌生的领域、架构决策——选一个更大的模型。当更小的模型无论你给多少上下文都自信地犯错时，你要的就是更大的模型。

更大的模型也更擅长处理歧义。而在更小的模型上，那些直接指挥执行的具体指令，才是更稳妥的成功之道。

当工作是常规性的——你能精确描述的改动、机械性的修改、关于上下文里已有代码的问题——就选更小的模型。没必要为任务用不上的能力付钱。

如果 Claude 已经有了所有相关的上下文，也明显努力过了，却还是搞错了；那就是一个信号，该选更大的模型。而如果你已经在更大的模型上，而且工作有一阵子都是常规性的，降一档会提升速度，通常还能降低成本，同时不影响输出质量。

### 努力程度：Claude 努力得不够

如果 Claude 是因为努力得不够而搞错——跳过了一个文件、没跑测试，或者没反复核对它的工作——那就选更高的努力程度。如果你之前选了一个低于模型默认值的努力程度，这一点最相关。

## 专才、专家和通才

我喜欢这样去想这两个设置：Fable 是一个专才，能处理几乎没人能处理的问题；Opus 是专家；Sonnet 则是一个非常优秀的通才。而努力程度，决定了他们当中任何一个在你的任务上花多少时间。

低努力程度下的 Opus，就像你和一位对你这类问题有深厚经验的专家，争取到了五分钟。他们带来的是你代码库里哪儿都没有的知识：他们见过的模式、他们知道要去查的坑，以及那种只有靠解决过大量类似问题才能积攒下的经验。但这五分钟意味着，只是快速浏览一遍你的代码，而不是仔仔细细地把每个文件过一遍。

高努力程度下的 Sonnet，是一个有整整一下午时间的通才。他们很擅长写代码，而且会把所有东西都读一遍、把东西跑起来、反复核对工作，最终彻底理解你这份特定的代码。

Fable 是你当所有人都卡住时才会去叫的那个专才。即使在低努力程度下，他们也能发现别人都发现不了的那个点。这种识别能力，也正是你花钱最多的地方，所以值得把它留给那些真正需要它的任务。

它们当中没有哪一个是普遍"更好"的。模型这个设置，粗略地说就是能力有多强；努力程度这个设置，粗略地说就是有多彻底。大多数真实任务，两者都需要一些。

## 努力程度、模型和 token 消耗

那么，模型选择、努力程度和 token 消耗，这三者是怎么相互作用的？这取决于任务。

在同样的努力程度下做常规工作时，更大的模型和更小的模型通常都能做对。更大的模型会消耗更多 token 在额外的验证步骤上，而且每个 token 的单价更高。这就是为什么在常规段落里降到更小的模型，能在不牺牲质量的情况下省下真金白银。

![曲线仅供示意，展示的是一个简单到两个模型都能快速完成的任务。它们不代表真实的基准数据](https://img.riba2534.cn/images/2026/07/cc-model-effort-10-curve-simple.png)

而在更难、多步的工作上，这个等式就翻转了。更小的模型不得不朝着自己能力的极限去硬磨，烧掉一次次迭代，而更大的模型用更少的步骤就达到了同样的质量门槛。

你为更大的模型在每个 token 上付了更多钱，但在那些真正让小模型吃力的任务上，每个任务的总成本反倒可能更低。更重要的是：更大的模型能够完成更小的模型完不成的任务，哪怕是在最高的努力程度下。

这一点在 Fable 身上最为明显。在长的、多步的工作上，它领先得最远。在我们的测试里，它完成了 Opus 和 Sonnet 在任何努力程度下都够不到的工作。它每个 token 的成本也最高，这是另一个把它留给真正需要的工作的理由。

![曲线仅供示意，展示的是一个困难到足以考验两个模型的任务。它们不代表真实的基准数据](https://img.riba2534.cn/images/2026/07/cc-model-effort-11-curve-hard.png)

上面这些图里关键的一点是：努力程度决定的是 Claude 愿意沿着曲线走多远。但这并不意味着，Claude 就需要走那么远才能完成任务。

最后，努力程度塑造 token 消耗，但并不限制它。系统里唯一的硬上限是 `max_tokens`，当撞上它时会在输出途中把响应截断，但这是个粗暴的工具，主要和 API 开发者相关。更软的控制——比如任务预算，或者在你的提示词里要求 Claude 说简短点——会更有用。它们是模型经过训练会去遵循的引导（它会在接近上限时着手收尾），而不是一堵它会撞上去的墙。

---

努力程度改变的是 Claude 做多少工作。模型改变的是 Claude 知道什么。

当你对结果不满意时，在动这两个设置中的任何一个之前，先检查上下文：给 Claude 一个清晰的提示词、对的工具和技能，以及一种验证它自己工作的方式。

如果 Claude 还是搞错了，就问自己：它是懂得不够，还是努力得不够？**懂得不够，是模型的问题；努力得不够，是努力程度的问题。**
