上周，我们在 Claude Code 中发布了动态工作流（dynamic workflows）。现在，Claude 可以即时编写属于自己的骨架（harness），为手头的任务量身定制。

Claude Code 默认的骨架是为编程而生的，但它对许多其他类型的任务同样有用——因为事实证明，很多任务本身就和编程任务很像。不过，有那么几类任务，我们必须在 Claude Code 之上搭建定制骨架才能把性能压榨到极致，比如深度调研、安全分析、Agent 团队，或者代码审查。

**工作流让你能够动态地创建这些骨架**，使 Claude 能在 Claude Code 内部原生地解决上述所有问题，乃至更多。你还可以把这些工作流分享给别人、供他人复用。

在这篇文章里，我会分享自己使用工作流的初步体验和心得，好让你能充分发挥它的威力。

话虽如此，最佳实践仍在不断摸索中！**动态工作流往往会消耗更多 token，所以要仔细斟酌何时用、怎么用。**

## 示例提示词

在深入技术细节之前，我想先给出一些示例提示词，帮你打开思路，想象工作流能带来的种种可能：

- "这个测试大概每跑 50 次会失败 1 次。建一个工作流来复现它，提出若干假设，并在 worktree 里对这些假设做对抗式验证。`/goal` 在某个假设被验证成立之前不要停。"

- "用一个工作流，翻一遍我最近的 50 次会话，从中挖出我反复在做的纠正，把那些反复出现的提炼成 CLAUDE.md 规则。"

- "用一个工作流，把 Slack 里 #incidents 频道过去半年的内容挖一遍，找出那些反复出现、却没人开过工单的根因。"

- "拿我的商业计划书，跑一个工作流，让不同的 agent 分别从投资人、客户和竞争对手的视角把它批得体无完肤。"

- "这里有一个装了 80 份简历的文件夹，用工作流按后端岗位给它们排序，并复核前十名。用 AskUserQuestion 工具来问我，帮我定出一套评分标准。"

- "我得给这个 CLI 工具起个名。用工作流头脑风暴出一堆候选，再跑一场锦标赛挑出前三名。"

- "用工作流把我们代码里所有的 User 模型重命名成 Account。"

- "把我的博客草稿过一遍，用工作流对照代码库逐条核实每一个技术论断——我可不想把任何错误的东西发出去。"

## 动态工作流是怎么运作的

动态工作流会执行一个 JavaScript 文件，里面有几个特殊函数，专门用来派生和协调子 agent：

![动态工作流的核心函数：agent、parallel、pipeline](https://img.riba2534.cn/images/2026/06/final_image1.png)

动态工作流里同样可以用 JSON、Math、Array 这些标准 JavaScript 函数来处理数据。

有一点特别值得了解：动态工作流可以决定某个 agent 用哪个模型，以及子 agent 是否在各自独立的 worktree 中运行——这让 Claude 能够按需选择智能水平和隔离程度。

如果工作流被打断了——比如用户主动中止，或者关掉了终端——重新恢复会话时，工作流会从上次中断的地方接着往下跑。

## 为什么要用动态工作流

当你让 Claude Code 默认骨架去做一个任务时，它得在同一个上下文窗口里既做规划又做执行。对许多编程任务来说，这样非常高效；但碰到那些长时间运行、大规模并行、或者高度结构化的对抗性任务时，它有时就会失灵。

原因在于，**Claude 在单个上下文窗口里处理一个复杂任务的时间越长，就越容易掉进几种特定的失败模式**：

- **Agent 偷懒（Agentic laziness）**：指 Claude 在一个特别复杂、多环节的任务还没做完时就停了下来，只完成了一部分就宣布大功告成——比如安全审查里 50 项只处理了 20 项。

- **自我偏好偏差（Self-preferential bias）**：指 Claude 倾向于偏爱自己得出的结果或发现，尤其是在被要求对照评分标准去核验或评判它们的时候。

- **目标漂移（Goal drift）**：指在多轮交互中，尤其是在上下文压缩（compaction）之后，逐渐偏离了最初的目标。每一次摘要压缩都是有损的，像边缘情况的需求、或者"不要做 X"这类约束，都可能在过程中丢失。

创建一个工作流能够对抗这些问题：它编排多个相互独立的 Claude，每个都有自己的上下文窗口，以及聚焦、隔离的目标。

## 动态工作流 vs 静态工作流

你可能此前用 Claude Agent SDK 或 `claude -p` 创建过静态工作流，把多个 Claude Code 实例协调到一起。

但静态工作流因为得照顾到所有边缘情况，通常写得比较通用。**有了 Claude Opus 4.8 和动态工作流，如今的 Claude 已经聪明到足以为你的具体场景量身编写一套定制骨架。**

# 使用动态工作流的几种实用模式

你只要让 Claude 做一个工作流就能开始用，或者用触发词"ultracode"来确保 Claude Code 一定会创建工作流。

但建立起一套关于动态工作流如何运作的心智模型，能帮你搞清楚什么时候该用它，以及怎样通过提示词去引导 Claude。

Claude 在构建工作流时，可能会用到、并组合使用以下几种常见模式：

![六种工作流模式](https://img.riba2534.cn/images/2026/06/final_image10.png)

**分类后行动（Classify-and-act）**

用一个分类器 agent 来判定任务的类型，然后根据任务把它路由到不同的 agent 或行为上。或者，在最后用一个分类器来决定输出。

**扇出后汇总（Fan-out-and-synthesize）**

把一个任务拆成许多更小的步骤，在每个步骤上跑一个 agent，然后把这些结果汇总起来。当小步骤数量很多，或者每个步骤都受益于一个干净、独立的上下文窗口（这样它们就不会相互干扰或交叉污染）时，这种模式尤其有用。汇总这一步是一道屏障（barrier）——它会等所有扇出的 agent 都完成，再把它们的结构化输出合并成一个结果。

**对抗式验证（Adversarial verification）**

对每一个派生出来的 agent，再派生一个独立的 agent，对照评分标准或判定准则，对它的输出做对抗式验证。

**生成后筛选（Generate-and-filter）**

围绕一个主题生成一批点子，然后按评分标准或通过验证去筛选它们，去掉重复项，只返回质量最高、经过检验的点子。

**锦标赛（Tournament）**

与其把工作分摊下去，不如让多个 agent 在同一件事上彼此竞争。派生 N 个 agent，每个都用不同的方法去尝试同一个任务。然后由提示词或模型借助一个评判 agent，对结果两两比较，直到决出胜者。

**循环直到完成（Loop until done）**

对于工作量未知的任务，循环地派生 agent，直到满足某个停止条件（不再有新发现，或者日志里不再有错误），而不是固定跑几遍就收手。

# 使用场景

不妨发挥想象力，去琢磨什么时候、用什么方式让 Claude Code 创建动态工作流。我发现，**工作流有时在非技术性工作上甚至更有用。**

## 迁移与重构

Bun 就是用工作流从 Zig 重写到 Rust 的。关于这件事是怎么做到的，你可以在 Jarred 的 X 长帖里读到更多。

关键在于把任务拆解成一系列需要逐一处理的步骤，比如调用点（callsites）、失败的测试、模块等等。为每一处修改在一个 worktree 里派生一个子 agent 去完成修复，再让另一个 agent 做对抗式审查，然后合并它们。可以考虑告诉 agent 不要使用消耗资源的命令，这样你就能在不耗尽机器资源的前提下把并行度拉到最大。

## 深度调研

我们在 Claude Code 里发布了一个用到动态工作流的深度调研技能（`/deep-research`）。具体来说，它会扇出多路网页搜索、抓取来源、对其中的论断做对抗式验证，再汇总成一份带引用的报告。

但你做这类调研未必只针对网页搜索。比如，可以让 Claude 从 Slack 的上下文里汇编一份状态报告，或者通过深入探索代码库来研究某个功能是怎么运作的。

## 深度核验

反过来，如果你有一份报告，想把它引用的每一条事实论断都核实并溯源，那你可能会想生成这样一个工作流：让一个 agent 找出所有的事实论断，然后为每一条派生一个子 agent 去逐一仔细核查。你还可以再安排一个验证 agent 去检查那个负责溯源的子 agent，确保它找到的来源质量过硬。

![深度核验工作流：论断提取器 → 论断核查员 → 来源审计员](https://img.riba2534.cn/images/2026/06/final_image2.png)

## 排序

你可能有一份列表，想按某种定性的尺度来排序，而你相信 Claude Code 擅长做这种评估——比如，把客服工单按 bug 的严重程度排序。但如果你想在一个提示词里排 1000 多行，质量会下降，而且根本塞不进上下文。换个做法：跑一场锦标赛，用一条由两两比较 agent 组成的流水线（**相对判断比绝对打分更可靠**），或者并行地分桶排名再合并。每一次比较都是它自己独立的 agent，所以那个确定性的循环负责维护赛程表，只有当前的比对顺序留在上下文里。

![排序锦标赛：1000 个条目两两对决直到决出排序](https://img.riba2534.cn/images/2026/06/final_image3.png)

## 记忆与规则遵守

如果你有一组特定的规则，发现 Claude 老是遗漏或者吃不准——哪怕已经写进了 CLAUDE.md——那就创建一个工作流，列出这些必须由验证 agent 来逐条检查的规则，一条规则配一个验证 agent。再造一个"怀疑论者"人设的子 agent 去复查这些规则、确保它们站得住脚，能帮你避免太多误报。

反过来也行：从你最近的会话和代码审查评论里，挖出你反复在做的那些纠正，用并行 agent 把它们聚类，对每个候选规则做对抗式验证（这条规则真能挡住一次真实的错误吗？），再把幸存下来的提炼回 CLAUDE.md。

![记忆与规则遵守：每条规则一个验证 agent，再交怀疑论者复核](https://img.riba2534.cn/images/2026/06/final_image8.png)

## 根因排查

调试时最有效的做法，是提出若干个相互独立的假设再逐一验证；但如果你只用一个上下文窗口，Claude 就可能撞上自我偏好偏差。

一个工作流能从结构上规避这个问题：它派生出多个 agent，分别从互不重叠的证据里生成假设。比如，日志、文件、数据各用一个独立的 agent。然后每个假设都要面对一组验证者和反驳者组成的评审团。

这不光适用于代码。工作流还能用于销售（三月份销售额为什么掉了？）、数据工程（这条数据管道为什么挂了？），或者任何复盘分析。

## 大规模分流处理

每个团队都有一条客服队列、一堆 bug 报告，或者某种人力没法完全消化的积压。

一个分流工作流会对每一项分类、与已经在跟踪的内容去重，然后采取行动。这可能意味着尝试修复，也可能意味着上报给人类用户。

分流工作流有一个好用的模式叫"隔离区"（quarantine）。它的做法是：禁止那些读取不可信公开内容的 agent 执行高权限操作，而把这些操作交给负责对信息采取行动的 agent 去做。

把分流工作流和 `/loop` 搭配起来，就能让 Claude 持续不断地做这件事。

![大规模分流：隔离区与可信区的权限隔离](https://img.riba2534.cn/images/2026/06/final_image6.png)

## 探索与品味

当你在探索一个解法的不同路径时，工作流会很有用——尤其是那些靠品味来判断的事情，比如设计或命名，而且能从一套评分标准里获益。

不妨让 Claude 探索出一堆方案，再给一个评审 agent 一套"什么样才算好方案"的评分标准。当评审 agent 觉得已经达标时，任务就算完成。方案也可以基于这套评分标准，通过锦标赛来排序或挑选。

## 评测

你可以为特定任务跑一些轻量级评测：在 worktree 里派生若干独立的 agent，再派生一些对比 agent，对照评分标准去比较并给具体的输出打分。比如，针对某套准则去评估、然后打磨你创建的某个技能。

## 模型与智能路由

创建一个针对你的任务调校过的分类器 agent，由它来决定该用哪个模型。当你的任务会涉及大量工具调用、且在执行前先做一番调研就能识别出最合适的模型时，这种做法会很有帮助。

举个例子，"解释 auth 模块是怎么工作的"这个任务最适合用哪个模型，取决于 auth 模块里有多少文件、以及代码库的形态。一个分类器 agent 可以先做这番调研，再根据任务的预期复杂度路由到 Sonnet 或 Opus。

## 什么时候不该用动态工作流

工作流还是个新东西。虽然有许多场景它能带来超额的回报，但并非每个任务都需要它，而且它可能最终消耗多得多的 token。

最好的用法，是创造性地用工作流把 Claude Code 推向你以前没尝试过的方向。对于常规编程任务，不妨问问自己：这真的需要更多算力吗？比如，**大多数传统编程任务并不需要一个 5 人评审团。**

# 构建动态工作流的技巧

**提示词**

为动态工作流写详细的提示词、用上我们前面讲的那些具体技巧，能产出最好的效果。

工作流不只适用于大任务。你可以提示模型用一个"快速工作流"。比如，你可以对某个假设快速做一次对抗式审查。

**与 /goal 和 /loop 搭配使用**

当你使用那些可以反复运行的工作流时，比如分流、调研或核验，把它们和 `/loop` 搭配起来定时运行，再用 `/goal` 设定一个硬性的完成要求。

**token 用量预算**

你可以给动态工作流设定明确的 token 用量预算，限制一个任务能用多少 token。你可以用类似"用 10k token"这样的预算去提示它，这会设定上限。

**保存与分享动态工作流**

在工作流菜单里按"s"就能保存工作流。你可以把它们提交进 `~/.claude/workflows`，或者通过技能（skill）来分发。

![工作流菜单：运行中与已完成的工作流及其用量](https://img.riba2534.cn/images/2026/06/final_image4.png)

要通过技能来分享，把你的 JavaScript 工作流文件放进技能的文件夹里，并在 SKILL.md 中引用它们。为了留出更大的灵活性，你或许会想提示 Claude：把技能里的工作流当成一个模板，而不是一个必须原封不动照跑的脚本。

![把工作流文件放进技能文件夹，在 SKILL.md 中引用](https://img.riba2534.cn/images/2026/06/final_image7.png)
