最成功的 Claude Code 部署，在配置、工具链和组织结构上都共享一组可被辨认出来的模式。本文是 **Claude Code at scale** 的一部分——这是一个新系列，覆盖工程组织在企业级规模下用 Claude Code 构建的最佳实践。

Claude Code 正在生产环境中运行：数百万行的 monorepo、有几十年历史的遗留系统、横跨几十个仓库的分布式架构，以及拥有数千名开发者的组织。这些环境带来的挑战，是更小、更简单的代码库不会遇到的——无论是每个子目录都不一样的构建命令，还是散落在一堆没有共同根目录的文件夹里的遗留代码。

本文覆盖的，是我们观察到的、那些促成 Claude Code 在规模化场景下被成功采用的模式。我们用"大型代码库"来指代相当宽泛的一类部署：数百万行的 monorepo、历经数十年构建起来的遗留系统、分散在多个独立仓库里的几十个微服务，或者上述情况的任意组合。这也包括那些跑在团队通常不会和 AI 编码工具联想到一起的语言上的代码库，比如 C、C++、C#、Java、PHP。（在这些情况下，Claude Code 的表现比大多数团队预期的要好，尤其是在最近几次模型发布之后。）尽管每一个大型代码库的部署都由它各自特定的版本控制、团队结构和长期积累下来的约定所塑造，但本文所讲的这些模式在它们之间是通用的，对于正在考虑采用 Claude Code 的团队来说是一个很好的起点。

## Claude Code 如何在大型代码库中导航

Claude Code 导航一个代码库的方式，和一名软件工程师会做的一样：它遍历文件系统、读取文件、用 grep 精确找到它需要的东西，并在整个代码库中追踪引用。它在开发者的机器上本地运行，不需要构建、维护一个代码库索引，也不需要把索引上传到服务器。

由 RAG 驱动的 AI 编码工具，其工作方式是把整个代码库做 embedding，并在查询时检索相关的片段。在大规模场景下，这类系统可能会失效，因为 embedding 流水线跟不上活跃的工程团队。等开发者去查询索引时，索引反映的还是这个代码库在数周、数天、甚至数小时之前的样子。于是检索会返回一个团队两周前就改了名的函数，或者引用一个上个 sprint 已经被删掉的模块——而且没有任何迹象表明这两者已经过期了。

Agentic 搜索避开了这些失败模式。当成千上万的工程师在提交新代码时，不存在需要维护的 embedding 流水线或中心化索引。每个开发者的实例都基于实时的代码库工作。

但这套方法有一个权衡：它在 Claude 拥有足够的起始上下文、知道该去哪里看的时候，效果最好。这意味着 Claude 导航的质量，由代码库被设置得有多好所塑造——通过 CLAUDE.md 文件和 skills 来分层地铺设上下文。如果你让它在一个十亿行的代码库里找出某个模糊模式的所有实例，那么活儿还没开始你就会先撞上上下文窗口的上限。在代码库设置上投入的团队，会看到更好的结果。

## harness 与模型同等重要

关于 Claude Code，最常见的误解之一，是认为它的能力完全由所使用的模型决定。团队会盯着一个模型的 benchmark、以及它在测试任务上的表现。但在实践中，**围绕模型构建起来的那套生态——也就是 harness——比模型本身更能决定 Claude Code 的表现**。

harness 由五个扩展点构成——CLAUDE.md 文件、hooks、skills、plugins，以及 MCP servers——每一个都承担不同的功能。团队构建它们的顺序很重要，因为每一层都建立在前一层之上。另外还有两项能力——LSP 集成和 subagents——把整套设置补全。下面我们逐一解释这些组件和能力各自做什么：

**CLAUDE.md 文件排在第一位。** 它们是 Claude 在每个 session 开始时自动读取的上下文文件：根目录文件给出大局，子目录文件给出本地约定。它们给 Claude 提供做好任何事情所需要的代码库知识。因为它们在每个 session 都会加载，不管任务是什么，所以让它们聚焦于那些普遍适用的内容，才能避免它们成为性能上的拖累。

**Hooks 让整套设置具备自我改进的能力。** 大多数团队把 hooks 想成"阻止 Claude 做错事的脚本"，但它们更有价值的用法是持续改进。一个 stop hook 可以在上下文还新鲜的时候，回顾这个 session 里发生了什么，并提议对 CLAUDE.md 的更新。一个 start hook 可以动态地加载团队特定的上下文，这样每个开发者无需手动配置就能为自己的模块拿到正确的设置。对于像 lint 和格式化这样的自动化检查，hooks 以确定性的方式强制执行规则，会比指望 Claude 记住一条指令产出更一致的结果。

**Skills 让正确的专业知识保持按需可用，又不会让每个 session 变得臃肿。** 在一个有几十种任务类型的大型代码库里，并不是所有专业知识都需要出现在每个 session 中。Skills 通过 progressive disclosure（渐进式披露）解决这个问题：把那些原本会争抢上下文空间的专门工作流和领域知识卸载出去，只在任务需要它们时才加载。比如，一个安全审查 skill 会在 Claude 评估代码漏洞时加载，而一个文档处理 skill 会在代码发生变更、文档需要更新时加载。

Skills 也可以被限定到特定路径，这样它们只在代码库的相关部分激活。一个拥有支付服务的团队，可以把他们的部署 skill 绑定到那个目录，这样当有人在 monorepo 的其它地方工作时，它就绝不会被自动加载。

**Plugins 把有效的做法分发出去。** 大型代码库的一个挑战是：好的设置可能会停留在小圈子里（tribal）。一个 plugin 把 skills、hooks 和 MCP 配置打包进一个单一的、可安装的包，这样当一名新工程师在第一天安装这个 plugin 时，他立刻就拥有了和那些一直在用 Claude 的人一样的上下文和能力。plugin 的更新可以通过受管的 marketplace 分发到整个组织。

举个例子，一家与我们合作的大型零售组织，构建了一个把 Claude 连接到他们内部分析平台的 skill，让业务分析师无需离开自己的工作流就能拉取业绩数据。在向整个业务部门广泛推广之前，他们以 plugin 的形式分发了它。

**语言服务器协议（LSP）集成，给 Claude 提供开发者在自己 IDE 里拥有的同样的导航能力。** 大多数大型代码库的 IDE 已经在跑一个 LSP，支撑着"跳转到定义"和"查找所有引用"。把这个能力暴露给 Claude，给了它符号级别的精度：它可以顺着一个函数调用追到它的定义、跨文件追踪引用，并区分不同语言中同名的函数。没有它，Claude 只能在文本上做模式匹配，可能会落到错误的符号上。一家与我们合作的企业软件公司，在他们的 Claude Code 推广之前就在全组织范围内部署了 LSP 集成，专门是为了让 C 和 C++ 的导航在规模化场景下可靠。对于多语言代码库，这是回报最高的投资之一。

**MCP servers 扩展一切。** MCP servers 是 Claude 连接到那些它本来够不到的内部工具、数据源和 API 的方式。最成熟的团队构建了 MCP servers，把结构化搜索暴露成一个 Claude 可以直接调用的工具。其他团队则把 Claude 连接到内部文档、工单系统或分析平台。

**Subagents 把探索与编辑分开。** 一个 subagent 是一个隔离的 Claude 实例，拥有它自己的上下文窗口；它接下一个任务、完成工作，并只把最终结果返回给父 agent。一旦 harness 就位，一些团队会起一个只读的 subagent 去映射一个子系统、把发现写进一个文件，然后让主 agent 在掌握全貌的情况下进行编辑。

下面这张表总结了每个组件做什么、它何时加载，以及我们在每个组件上看到的最常见的误区：

| 组件 | 是什么 | 何时加载 | 最适合 | 常见误区 |
|------|--------|---------|--------|---------|
| **CLAUDE.md** | Claude 自动读取的上下文文件 | 每个 session | 项目特定约定、代码库知识 | 把本该放进 skill 的可复用专业知识用它来承载 |
| **Hooks** | 在关键时刻运行的脚本 | 由事件触发 | 自动化一致的行为、捕获 session 中的经验 | 对那些本该自动运行的事情用 prompt 去处理 |
| **Skills** | 针对特定任务类型打包好的指令 | 按需，相关时加载 | 跨 session、跨项目复用的专业知识 | 把所有东西都塞进 CLAUDE.md 里，而不是做成 skill |
| **Plugins** | 打包好的 skills、hooks、MCP 配置 | 配置后始终可用 | 把一套有效的设置分发到整个组织 | 让好的设置停留在小圈子里 |
| **语言服务器协议（LSP）**\* | 通过语言专属服务器提供的实时代码智能 | 配置后始终可用 | 类型化语言中的符号级导航与自动报错检测 | 想当然地以为它会自动生效 |
| **MCP servers** | 与外部工具和数据的连接 | 配置后始终可用 | 让 Claude 访问它本来够不到的内部工具 | 在基础还没跑通之前就先去搭 MCP 连接 |
| **Subagents**\* | 为特定任务而起的独立 Claude 实例 | 被调用时 | 把探索与编辑分开、并行作业 | 在同一个 session 里既做探索又做编辑 |

> \* LSP 通过 plugin 层来访问；subagents 是一种委派能力，而不是一个需要配置的扩展点。

> 注：原文此处配有一张题为 **"The Claude Code harness"** 的时间线信息图，横轴为一次 session 的时间（Start → End），纵向逐行展示各组件的活跃节奏——CLAUDE.md（地基）从 session 早期起持续到结束；Hooks（自我改进）在开头和结尾各触发一次；Skills（渐进式披露）是散布的离散点；Plugins（分发）在 session 之前装好、全程在线；LSP（导航）和 MCP servers（扩展）是贯穿过程的离散点；Subagents（探索 + 编辑）是 session 中段单独切出去的一整段，标注为"映射 → findings 文件"。

## 来自成功部署的三个配置模式

你如何为一个大型代码库配置 Claude Code，在很大程度上取决于这个代码库本身是怎么组织的。尽管如此，在我们观察到的部署里，有三个模式始终一致地出现。

### 在规模化场景下让代码库可被导航

Claude 在一个大型代码库中能提供多少帮助，被它"找到正确上下文"的能力所框定。每个 session 加载太多上下文会降低性能，而上下文太少则会让 Claude 盲目导航。最有效的部署会在前期投入，把代码库做得对 Claude 可读。有几个模式始终一致地出现：

- **让 CLAUDE.md 文件保持精简且分层。** 当 Claude 在代码库中移动时，它会叠加式地加载这些文件：根目录文件给出大局，子目录文件给出本地约定。根目录文件应当只放指针和关键的坑（gotchas）；其它一切都会漂移成噪声。
- **在子目录中初始化，而不是在仓库根目录。** 当 Claude 被限定到与任务真正相关的那部分代码库时，它工作得最好。在 monorepo 里这可能让人觉得反直觉，因为工具链常常假设拥有根访问权限；但 Claude 会自动沿着目录树往上走，并加载它沿途找到的每一个 CLAUDE.md 文件，所以根级别的上下文永远不会丢失。
- **按子目录限定 test 和 lint 命令。** 当 Claude 只改动了一个服务，却去跑完整的测试套件，会造成超时，并把上下文浪费在不相关的输出上。子目录级别的 CLAUDE.md 文件应当指明适用于这部分代码库的命令。这对于面向服务的代码库效果很好——在那里每个目录都有自己的 test 和 build 命令。在有深度跨目录依赖的、编译型语言的 monorepo 里，按子目录限定更难做到，可能需要项目特定的构建配置。
- **用 `.ignore` 文件来排除生成的文件、构建产物和第三方代码。** 把 `permissions.deny` 规则提交进 `.claude/settings.json`，意味着这些排除是版本受控的，于是团队里每个开发者都能得到同样的降噪，而无需自己去配置。在某些代码库里，生成的文件本身就是开发工作的对象。从事代码生成器工作的开发者，可以在他们的本地设置里覆盖项目级别的排除，而不影响团队其他人。
- **当目录结构本身做不到这件事时，构建代码库地图。** 对于那些代码没有被整合进常规目录结构的组织，在仓库根目录放一个轻量的 markdown 文件，列出每一个顶层文件夹并配一句话描述里面装的是什么，就等于给了 Claude 一份它在打开文件之前可以扫一眼的目录。对于有数百个顶层文件夹的代码库，这件事最好用分层的方式来做：根目录文件只描述最高层级的结构，子目录的 CLAUDE.md 文件提供下一层级的细节，并随着 Claude 在目录树中移动而按需加载。对于更简单的情况，用 @ 提及 Claude 应当参考的具体文件或目录，也能起到同样的作用。
- **运行 LSP servers，让 Claude 按符号搜索，而不是按字符串搜索。** 在一个大型代码库里 grep 一个常见的函数名，会返回数千个匹配，Claude 会烧掉上下文去逐个打开文件、判断哪个才重要。LSP 只返回那些指向同一个符号的引用，因此过滤发生在 Claude 读取任何东西之前。设置它需要为你的语言安装一个代码智能 plugin 以及对应的语言服务器二进制文件；Claude Code 文档里覆盖了可用的 plugin 和故障排查。

> 一个需要注意的点：存在一些边缘情况，即便是分层的 CLAUDE.md 这套做法也会失效，比如有数十万个文件夹和数百万个文件的代码库，或者跑在非 git 版本控制上的遗留系统。我们会在本系列后续的文章里处理它们的挑战。

### 随着模型智能演进，主动维护 CLAUDE.md 文件

随着模型演进，为你当前模型写的指令，可能会反过来对一个未来的模型起反作用。那些曾经引导 Claude 走过它过去不擅长的模式的 CLAUDE.md 文件，等下一个模型发布时，可能要么变得不再必要，要么变成主动的束缚。举个例子，一条告诉 Claude 把每一次重构都拆成单文件改动的 CLAUDE.md 规则，可能曾帮助一个较早的模型保持在正轨上，但会阻止一个更新的模型去做它本来处理得很好的、协调式的跨文件编辑。

那些为补偿特定模型局限而构建的 skills 和 hooks——无论这些局限是在模型的推理上，还是在 Claude Code 自身的工具上——一旦那些局限不复存在，就变成了额外开销。举个例子，一个为了在 Perforce 代码库里强制执行 `p4 edit` 而拦截文件写入的 hook，在 Claude Code 加入了原生 Perforce 模式之后，就变得多余了。

团队应当预期：每三到六个月做一次有意义的配置回顾；此外，每当在重大模型发布之后感觉性能像是停在了原地，也值得做一次。

### 为 Claude Code 的管理与采用分配 ownership

光有技术配置并不能驱动采用。那些做对了的组织，也在组织层面进行了投入。

传播得最快的那些推广，在广泛开放之前就有一笔专门的基础设施投入。一个小团队，有时甚至只有一个人，把工具链接好，让 Claude 在开发者第一次接触它时就已经契合了开发者的工作流。在一家公司，几名工程师构建了一套 plugins 和 MCPs，第一天就可用。在另一家公司，一整个专注于管理 AI 编码工具的团队，在推广开始之前就把基础设施准备就绪。两种情况下，开发者的第一次体验都是高效的，而不是令人沮丧的，采用也从这里扩散开来。

| 阶段 | 名称 | 时点 | 发生了什么 |
|------|------|------|-----------|
| **Phase 01** | 静默投入（Quiet investment） | 广泛开放之前 | owner 一块一块地把基础设施拼装起来 |
| **Phase 02** | 推广落地（The rollout lands） | 第一天 | 基础设施就绪，第一波开发者发现它确实高效 |
| **Phase 03** | 采用扩散（Adoption spreads） | 推广之后 | 口碑在团队之间传开，harness 上的"人口"增长 |

> 注：上表对应原文题为 **"Three phases of a Claude Code rollout"** 的信息图。

今天做这项工作的团队，往往挂在 developer experience 或 developer productivity（开发者体验/开发者效能）之下——这通常正是负责新工程师 onboarding 和构建开发者工具的那个职能。一个在好几个组织里正在浮现的新角色是 **agent manager**：一个 PM/工程师混合的职能，专门负责管理 Claude Code 生态。对于没有专门团队的组织，最小可行版本是一个 **DRI**：一个对 Claude Code 配置拥有 ownership 的人，他有权对设置、权限策略、plugin marketplace 和 CLAUDE.md 约定做出决断，并负责让它们保持最新。

自下而上的采用会产生热情，但如果没有人去把"有效的做法"中心化，它可能会碎片化。你需要有一个人或一个团队，去组装并布道正确的 Claude Code 约定（比如一套标准化的 CLAUDE.md 层级，或一组精心挑选的 skills 和 plugins）。没有这项工作，知识会停留在小圈子里，采用会陷入停滞。

在大型组织里，尤其是那些处于受监管行业的，治理问题会很早就出现，比如：谁来控制哪些 skills 和 plugins 可用，你如何防止数千名工程师各自独立地重复造同一个轮子，你如何确保 AI 生成的代码走和人写的代码同样的审查流程？为了尽早处理这些问题，我们建议从一组定义好的、被批准的 skills，必需的代码审查流程，以及有限的初始访问开始，再随着信心建立而扩大。

我们观察到，最顺畅的部署发生在那些早早建立跨职能工作组的组织里——它们把工程、信息安全和治理方面的代表聚到一起，共同定义需求，并构建一份推广路线图。

## 把这些模式应用到你的组织

Claude Code 是围绕传统的软件工程环境设计的：在那里，工程师是代码库的主要贡献者，仓库使用 Git，代码遵循标准的目录结构。大多数大型代码库符合这个模子，但一些非传统的设置需要额外的配置工作，比如带有大体量二进制资源的游戏引擎、使用非常规版本控制的环境，或者由非工程师向代码库贡献内容。我们的指南假设的是一套传统的设置，而我们所描述的这些模式，已经在我们许多客户那里行之有效。剩下的任何复杂性，都需要针对你自己的代码库、工具链和组织做出具体的判断。这正是 Anthropic 的 Applied AI 团队直接与工程团队合作、把这些模式翻译成你的组织具体需求的地方。

下面是原文给出的**起步清单**——在你第一次广泛推广之前要搭好的六件事：

- [ ] **梳理你的 CLAUDE.md 文件** —— 根目录文件放全组织通用的上下文，子目录放团队特定规则，每个文件约 200 行。
- [ ] **为不可妥协项接上 hooks** —— 格式化、测试命令、安全检查，每次都强制执行，不是可选项。
- [ ] **为领域专业知识安装 skills 和 plugins** —— 那些不属于 CLAUDE.md 指令的可复用知识。
- [ ] **让代码库可被导航** —— `.ignore` 文件、LSP servers，以及代码库地图——更少噪声，更好结果。
- [ ] **安排定期的配置回顾** —— 每 3-6 个月一次，或在新模型发布时。旧的 workaround 会变成额外开销。
- [ ] **分配 ownership** —— 至少要有一个人，拥有 CLAUDE.md 层级并负责让它保持最新。
