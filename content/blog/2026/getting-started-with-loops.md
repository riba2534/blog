最近关于"设计循环"而非直接给你的编程 agent 写提示词的讨论很多。如果你花点时间在 X 上试图搞清楚到底什么是循环，你会碰到好几种不同的答案。

在 Claude Code 团队，**我们把循环定义为 agent 重复执行工作周期，直到满足停止条件为止。**我们根据以下几个维度对循环进行分类：

- 循环如何被触发
- 循环如何停止
- 使用了哪个 Claude Code 原语
- 每种循环最适合什么类型的任务

本文将介绍主要的循环类型、何时使用每种类型，以及如何在使用过程中保持代码质量并管理 token 用量。**并非所有任务都需要复杂的循环；从最简单的方案开始，有选择地使用这些模式。**

## 基于回合的循环

![基于回合的循环](https://img.riba2534.cn/images/2026/07/cn_02_turn_based.png)

- **触发方式**：用户提示词。
- **停止条件**：Claude 判断任务已完成或需要更多上下文。
- **最适合**：不属于常规流程或计划的较短任务。
- **控制用量**：编写具体的提示词，并使用 skills 改进验证流程以减少回合数。

你发送的每条提示词都会启动一个手动循环，由你在每个回合中主导。Claude 收集上下文、执行操作、检查自己的工作、必要时重复，然后回复。我们称之为 agentic loop（智能体循环）。

举个例，让 Claude 创建一个点赞按钮。它读取你的代码，做出修改，运行测试，然后交还它认为能用的成果。接下来你手动检查工作，再写下一段提示词。

你可以通过将手动验证步骤编码为 SKILL.md 来改进验证环节，**这样 Claude 就能端到端地自行检查更多工作。**这应该包含能让 Claude 观察、衡量或与结果交互的工具或连接器。检查越量化，Claude 就越容易自我验证。

例如，在你的 SKILL.md 文件中可以这样写：

```markdown
---
name: verify-frontend-change
description: 在声明完成之前，端到端地验证任何 UI 变更。
---

# 验证前端变更
永远不要仅凭一次成功的编辑就报告 UI 变更已完成。像人工 reviewer 那样去验证：

1. 启动开发服务器，在浏览器中打开编辑过的页面。

2. 直接与变更交互。对于新的控件（按钮、输入框、开关）：点击它，确认预期的状态变化，并截取前后对比截图。

3. 检查浏览器控制台：零新增错误或警告。

4. 使用 Chrome Devtools MCP，运行性能追踪并审计 Core Web Vitals。

如果任何一步失败，修复问题并从第 1 步重新运行——不要交回部分验证过的成果。
```

## 基于目标的循环（/goal）

![基于目标的循环](https://img.riba2534.cn/images/2026/07/cn_03_skill_example.png)

- **触发方式**：实时手动提示词。
- **停止条件**：目标达成或达到最大回合数。
- **最适合**：具有可验证退出标准的任务。
- **控制用量**：设定明确的完成标准和显式的回合上限，比如"5 次尝试后停止"。

有时候单回合不够，尤其是对更复杂的任务而言。**agent 在能迭代时会表现更好。**你可以通过 `/goal` 定义"完成"是什么样子，从而延长 Claude 持续迭代的时间。

当你定义了成功标准后，Claude 就不需要自行判断什么是"足够好"并提前终止循环。每次 Claude 试图停止时，**一个评估模型会检查你的条件，并把它送回继续工作**，直到目标达成或达到你定义的回合上限。

这就是为什么确定性标准（比如通过的测试数量或达到某个分数阈值）如此有效。

例如：

```bash
/goal 将首页 Lighthouse 评分提升至 90 或以上，5 次尝试后停止。
```

## 基于时间的循环（/loop 和 /schedule）

- **触发方式**：指定的时间间隔。
- **停止条件**：你取消它，或工作完成（PR 合并、队列清空）。
- **最适合**：周期性工作，或与外部环境/系统对接的场景。
- **控制用量**：设置更长的间隔，或基于事件而非时间来响应。

有些 agent 工作是周期性的：任务本身不变，只是输入在变。比如每天早上总结 Slack 消息。另一些工作依赖于外部系统，最简单的对接方式就是按间隔检查并响应变化。比如一个可能收到 code review 或 CI 失败的 PR。

对于这些场景，你可以用 `/loop` 让 Claude 按间隔重复执行一段提示词。例如：

```bash
/loop 5m 检查我的 PR，处理 review 意见，修复失败的 CI
```

`/loop` 在你的电脑上运行，所以关掉它就停了。你可以通过 `/schedule` 创建一个例行任务，将循环搬到云端运行。

## 主动式循环

![主动式循环](https://img.riba2534.cn/images/2026/07/cn_04_proactive.png)

- **触发方式**：事件或计划触发，没有实时人工参与。
- **停止条件**：每个任务在目标达成时退出。例行任务本身一直运行，直到你关闭它。
- **最适合**：定义清晰的周期性工作流：bug 报告、issue 分类、迁移、依赖升级等。
- **控制用量**：将例行任务路由到更小、更快的模型，将判断决策留给能力最强的模型。

上述原语，配合其他 Claude Code 功能（如 auto mode 和 dynamic workflows（研究预览版）），可以组合成一个用于长时间运行工作的循环。

以处理用户反馈为例，你可以这样组合：

1. 用 `/schedule`（研究预览版）运行检查新报告的例行任务
2. 用 `/goal` 定义"完成"的标准，用 skills 记录验证方式
3. 用 dynamic workflows 编排多个 agent，分别分类、修复和审查每个报告
4. 开启 auto mode，让例行任务不停下来询问权限

组合在一起后，一段提示词大概长这样：

```bash
/schedule 每小时：检查 project-feedback 频道中的 bug 报告。
/goal：本次运行中发现的所有报告都完成分类、处理和回复之前不要停止。
修复 bug 时，使用 workflow 在三个并行的 worktree 中探索解决方案，
并由一个 judge 对它们进行对抗性审查。
```

## 保持代码质量

循环产出的质量取决于围绕它的系统。在设计系统时：

- **保持代码库本身整洁**：Claude 会遵循你代码库中已有的模式和约定。
- **给 Claude 一个验证自己工作的方式**：用 skills 把你和你的团队认为的"好"编码下来。
- **让文档容易获取**：框架和库的文档包含最新的最佳实践。
- **用第二个 agent 做 code review**：**一个具有全新上下文的 reviewer 偏见更少，也不会受主 agent 推理过程的影响。**你可以使用内置的 `/code-review` 技能或 GitHub Code Review。

当某个结果达不到标准时，不要止步于修复这一个问题——试着把它编码进系统，让所有未来的迭代都受益。

## 管理 Token 用量

为了管理 token 用量，循环应该有清晰的边界：

- **为任务选择合适的原语和模型**：小任务不需要多个 agent 或循环。有些任务可以用更便宜、更快的模型。
- **定义清晰的成功和停止条件**：具体说明"完成"是什么样子，让 Claude 能更快（但不会太快）得出解决方案。
- **大规模运行前先试点**：dynamic workflows 可能产生数百个 agent。先在较小的工作切片上评估用量。
- **对确定性工作使用脚本**：运行脚本比逐步推理更便宜。比如一个 PDF skill 可以附带一个填表脚本，Claude 每次直接运行，而不是重新推导代码。
- **不要比需要更频繁地运行例行任务**：让间隔与你所关注对象的变化频率匹配。
- **审查用量**：`/usage` 命令可以按 skills、subagent 和 MCP 拆分最近的用量；无参数运行 `/goal` 显示当前回合数和 token 用量；`/workflows` 显示每个 agent 的 token 用量，你可以随时停止某个 agent。

## 入门建议

总结一下：

| 循环类型 | 你交出去的是 | 适用场景 | 选用工具 |
|---------|------------|---------|---------|
| 基于回合 | 检查环节 | 你在探索或决策中 | 自定义验证 skills |
| 基于目标 | 停止条件 | 你清楚"完成"长什么样 | /goal |
| 基于时间 | 触发机制 | 工作在你的项目之外按计划到达 | /loop、/schedule |
| 主动式 | 提示词 | 工作是周期性的且定义清晰 | 以上全部，外加 dynamic workflows |

要开始使用循环，先看看你已经做的工作。**挑一件你是瓶颈的任务，问问自己哪部分可以交出去**：你能写出验证检查吗？目标够清晰吗？这项工作是否按计划到达？

有了想法之后，把循环跑起来，观察结果——比如在哪里卡住了、哪里过度执行了，**不要害怕反复迭代它。**

更多信息请阅读 Claude Code 文档中关于并行运行 agent 的部分，以及 loop、schedule、goal 和 dynamic workflows 相关页面。
