技能（skills）已经成为 Claude Code 里最常被用到的扩展点之一。它们灵活、好做，也方便分发。

但正是这份灵活，让人很难判断怎么用才最好。哪种技能值得做？一个技能该怎么组织结构？什么时候该把它分享给别人？

我们在 Anthropic 内部把技能用得很重，目前有数百个技能在活跃使用。下面这些，就是我们在用技能加速开发的过程中摸索出来的心得。

## 什么是技能

技能是一个个文件夹，里面装着指令、脚本和资源，agent 可以发现并使用它们，从而把事情做得更准、更高效。这篇文章默认你已经了解技能的基础知识；如果你是新手，可以先从我们在 Skilljar 上的《Agent 技能入门》课程开始。

我们常听到一个关于技能的误解，认为它们"不过是一些 markdown 文件"。它们其实是文件夹，里面可以包含脚本、素材、数据等等，agent 能去发现、探索并操作这些内容。

在 Claude Code 里，技能还有一大堆配置选项，包括注册动态 hook。

我们发现，**Claude Code 里那些最有效的技能，往往都把这些配置选项和文件夹结构用得很到位。**

## 技能的类型

把 Anthropic 内部所有技能盘点了一遍之后，我们注意到它们能聚成九个类别。最好的技能都能干净地落进其中某一类；那些想做太多事的技能，往往横跨好几类，把 agent 搞糊涂。这并不是一份权威清单，但它是个有用的框架，能帮你找出自己技能库里的空白。

![Claude Code 团队把内部技能做了分类，发现它们能归入九个不同的类别](https://img.riba2534.cn/images/2026/06/cc-skills-categories.png)

### 1. 库与 API 参考

这类技能讲清楚如何正确使用某个库、CLI 或 SDK。它们既可以针对内部库，也可以针对那些 Claude Code 偶尔会用不利索的常见库。这类技能通常会附带一个参考代码片段的文件夹，外加一份坑点清单，提醒 Claude 写脚本时绕开它们。

例子包括：

- `billing-lib` —— 你们内部的计费库：边缘情况、暗坑等等。
- `internal-platform-cli` —— 你们内部 CLI 封装的每一个子命令，并附上各自的使用时机示例。
- `sandbox-proxy` —— 为开发工作配置组织的出口网关：哪些主机可达、如何调试"connection refused"错误、如何添加白名单条目。

### 2. 产品验证

这类技能描述如何测试或验证你的代码是否正常工作。它们经常和 playwright、tmux 或其他外部工具搭配做验证。

在内部，验证类技能对 Claude 产出质量的影响是最可量化的。**让一位工程师花上一周时间，专门把你们的验证技能打磨到极致，是很值的。**

可以考虑一些做法，比如让 Claude 录一段它产出的视频，这样你就能确切看到它测了什么；又或者在每一步都对状态做程序化断言。这些通常是通过在技能里塞进各种脚本来实现的。

例子包括：

- `signup-flow-driver` —— 在无头浏览器里跑一遍注册 → 邮箱验证 → 引导上手，并在每一步都挂上断言状态的 hook。
- `checkout-verifier` —— 用 Stripe 测试卡驱动结账 UI，验证发票确实落到了正确的状态。
- `tmux-cli-driver` —— 用于交互式 CLI 测试，适合那些需要 TTY 才能验证的场景。

### 3. 数据获取与分析

这类技能连接你的数据栈和监控栈。它们可能包含带凭证拉取数据的库、特定的看板 ID 等等，以及关于常见工作流或取数方式的说明。

例子包括：

- `funnel-query` —— "我该 join 哪些事件才能看到注册 → 激活 → 付费"，外加那张真正持有规范 `user_id` 的表。
- `cohort-compare` —— 比较两个分群的留存或转化，标出统计显著的差异，并链接到对应的分群定义。
- `grafana` —— 数据源 UID、集群名、"问题 → 看板"的查找表。
- `datadog` —— 字段参考（`@request_id` 还是 `trace_id`）、服务列表、指标前缀约定。

### 4. 业务流程与团队自动化

这类技能把重复的工作流自动化成一条命令。它们通常是相当简单的指令，但可能对其他技能或 MCP 有更复杂的依赖。对这类技能来说，把过往结果存进日志文件，能帮模型保持一致性，并回顾此前几次的执行。

例子包括：

- `standup-post` —— 聚合你的工单系统、GitHub 活动和此前的 Slack 内容 → 格式化的站会播报，只保留增量。
- `create-<ticket-system>-ticket` —— 强制 schema（合法的枚举值、必填字段），外加创建后的工作流（@ 评审人、在 Slack 里贴链接）。
- `weekly-recap` —— 已合并的 PR + 已关闭的工单 + 部署记录 → 格式化的周报。

### 5. 脚手架与模板

这类技能为代码库里某个特定功能生成框架样板。你可以把这类技能和可组合的脚本搭配起来。当你的脚手架带有那种纯靠代码覆盖不了的自然语言需求时，它们尤其有用。

例子包括：

- `new-<framework>-workflow` —— 按你的注解脚手架出一个新的服务/工作流/handler。
- `new-migration` —— 你的迁移文件模板，外加常见坑点。
- `create-app` —— 一个全新的内部应用，认证、日志和部署配置都已预先接好线。

### 6. 代码质量与审查

这类技能在你的组织内部强制代码质量，并辅助代码审查。它们可以包含确定性的脚本或工具以追求最大的稳健性。你可能会想让这类技能作为 hook 的一部分自动运行，或放进 GitHub Action 里。

- `adversarial-review` —— 派生一个全新视角的子 agent 来挑刺、实施修复、反复迭代，直到发现的问题退化成吹毛求疵为止。
- `code-style` —— 强制代码风格，尤其是 Claude 默认做得不太好的那些风格。
- `testing-practices` —— 关于如何写测试、以及该测什么的说明。

### 7. CI/CD 与部署

这类技能帮你在代码库里拉取、推送和部署代码。它们可能会引用其他技能来收集数据。

例子包括：

- `babysit-pr` —— 盯着一个 PR → 重试不稳定的 CI → 解决合并冲突 → 启用自动合并。
- `deploy-<service>` —— 构建 → 冒烟测试 → 逐步放量并对比错误率 → 一旦出现回归就自动回滚。
- `cherry-pick-prod` —— 隔离的 worktree → cherry-pick → 解决冲突 → 按模板提 PR。

### 8. 运行手册（Runbooks）

这类技能拿到一个症状（比如一条 Slack 讨论串、一个告警或一个错误特征），走完一套多工具的排查流程，最后产出一份结构化的报告。

例子包括：

- `<service>-debugging` —— 为你流量最高的那些服务，把"症状 → 工具 → 查询模式"映射出来。
- `oncall-runner` —— 拉取告警 → 排查那些惯犯 → 格式化出一份结论。
- `log-correlator` —— 给定一个 request ID，从每一个可能经手过它的系统里拉出匹配的日志。

### 9. 基础设施运维

这类技能执行例行维护和运维操作，其中一些涉及破坏性动作，正好能从护栏中获益。它们让工程师在关键操作中更容易遵循最佳实践。

例子包括：

- `<resource>-orphans` —— 找出孤儿 pod/卷 → 发到 Slack → 静置一段时间 → 用户确认 → 级联清理。
- `dependency-management` —— 你们组织的依赖审批工作流。
- `cost-investigation` —— "我们的存储/出流量账单为什么突然飙升"，附上具体的桶和查询模式。

## 做技能的若干技巧

一旦你定下了要做哪个技能，接下来该怎么写？下面是 Claude Code 团队做技能的一些最佳实践、技巧和窍门。

### 不要陈述显而易见的事

![同一个计费库技能随时间演进：第 1 天太单薄，第 2 周开始补坑点，第 3 个月坑点更丰富](https://img.riba2534.cn/images/2026/06/cc-skills-gotchas.png)

Claude 本来就会写代码，也能读你的代码库。一个只是把 Claude 默认就会做的事重述一遍的技能，只增加了上下文，却没增加价值。如果你要发布的技能主要是关于知识的，那就聚焦在那些能把 Claude 推离它惯常思路的信息上。

前端设计技能就是个绝佳的例子；它由 Anthropic 的一位工程师打造，通过和客户反复打磨来改进 Claude 的设计品味，避开诸如 Inter 字体和紫色渐变这类老套路。

### 构建一个坑点清单

任何技能里信噪比最高的内容，就是坑点（Gotchas）这一节。这些章节应当从 Claude 使用你的技能时反复栽跟头的那些地方一点点攒起来。理想情况下，你会随着时间不断更新技能，把这些坑收纳进去。

例如：

> "subscriptions 表是只追加的。你想要的那一行是 version 最高的那行，不是 `created_at` 最近的那行。" "这个字段在 API 网关里叫 `@request_id`，在计费服务里叫 `trace_id`。它们是同一个值。" "即使 Stripe 的 webhook 实际没处理成功，staging 也会返回 200。要查 `payment_events` 才能知道真实状态。"

### 用好文件系统与渐进式披露

![SKILL.md 文件指向另外几个文件，Claude 可以在特定情况下去引用。比如任务挂起时，它就该去引用 stuck-jobs.md](https://img.riba2534.cn/images/2026/06/cc-skills-progressive.png)

正如我们前面说的，一个技能是一个文件夹，而不只是一个 markdown 文件。你应当把整个文件系统看作一种上下文工程和渐进式披露的形式。告诉 Claude 你的技能里有哪些文件，它就会在合适的时机去读它们。

渐进式披露最简单的形式，就是指向其他 markdown 文件让 Claude 使用。比如，你可以把详细的函数签名和用法示例拆进 `references/api.md`。

再举个例子：如果你的最终产出是一个 markdown 文件，你可以在 `assets/` 里放一个模板文件供它复制使用。

你可以有一个个装着参考资料、脚本、示例等内容的文件夹，帮助 Claude 更高效地干活。

### 别把 Claude 逼上独木桥

Claude 通常会尽量遵守你的指令，而正因为技能的复用性如此之强，你得当心别把指令写得太死。给 Claude 它需要的信息，但也给它根据情况临场调整的余地。

例如：

![过于死板的逐步指令 vs 更好的、保留意图的开放式指令](https://img.riba2534.cn/images/2026/06/cc-skills-railroading.png)

### 把初始设置想清楚

![上面这个技能写成：当 Slack 频道没有包含在配置里时，就提示用户](https://img.riba2534.cn/images/2026/06/cc-skills-setup.png)

有些技能可能需要用用户提供的上下文来做初始设置。比如，如果你在做一个把站会播报发到 Slack 的技能，你可能会想让 Claude 来问该发到哪个 Slack 频道。

做这件事有一个不错的模式：像上面的例子那样，把这些设置信息存进技能目录里的一个 `config.json` 文件。如果配置还没设好，agent 就可以向用户询问相关信息。

如果你想让 agent 抛出结构化的多选问题，可以指示 Claude 使用 AskUserQuestion 工具。

### 给模型写描述，而不是给人写

当 Claude Code 启动一个会话时，它会把每一个可用技能连同其描述列成一张清单。Claude 扫的就是这张清单，来判断"这个请求有对应的技能吗？"。这意味着 description 字段不是一段摘要，而是对何时触发该技能的描述。

![在描述里包含技能的触发词（比如"babysit"）会很有帮助](https://img.riba2534.cn/images/2026/06/cc-skills-descriptions.png)

### 帮 Claude 记住

![这个文本日志文件帮 Claude 记住过往的事件，比如审查过 Sarah 的 auth PR](https://img.riba2534.cn/images/2026/06/cc-skills-memory.png)

有些技能可以通过在自身内部存储数据，来包含某种形式的记忆。你可以把数据存进任何东西里——简单到一个只追加的文本日志文件或 JSON 文件，复杂到一个 SQLite 数据库。

比如，一个 `standup-post` 技能可能会保留一份 `standups.log`，记录它写过的每一条播报，这样下次你再运行它时，Claude 就能读到自己的历史，说出从昨天到现在有哪些变化。

你可以用环境变量 `${CLAUDE_PLUGIN_DATA}` 拿到一个稳定的目录来存数据，关于在技能里持久化数据，可以在[这里](https://code.claude.com/docs/en/plugins-reference#persistent-data-directory)读到更多。

### 存放脚本、生成代码

你能交给 Claude 的最强工具之一，就是代码。给 Claude 脚本和库，能让 Claude 把它的回合花在组合上——决定下一步做什么，而不是重新拼凑样板代码。

比如，在你的 data-science 技能里，你可能有一个从事件源拉数据的函数库。为了让 Claude 能做复杂分析，你可以给它一组这样的辅助函数：

![lib/signups.py：一组带文档字符串的取数辅助函数](https://img.riba2534.cn/images/2026/06/cc-skills-helpers.png)

这样，对于"周二发生了什么？"这类提示词，Claude 就能即时生成脚本，把这些功能组合起来，做更进阶的分析。

![investigate.py（由 Claude 生成）：组合上面的辅助函数定位异常](https://img.riba2534.cn/images/2026/06/cc-skills-scripts.png)

### 用按需挂载的 hook

技能可以包含一些 hook，它们只在技能被调用时才激活，并且只在该会话期间有效。把它用在那些更"有主见"的 hook 上——那些你不想让它一直跑、但偶尔又极其有用的 hook。

例如：

- `/careful` —— 通过 Bash 上的 PreToolUse 匹配器，拦截 `rm -rf`、`DROP TABLE`、force-push、`kubectl delete`。你只想在明知自己正在动生产环境时才用它——要是一直开着，会把你逼疯。
- `/freeze` —— 拦截任何不在指定目录里的 Edit/Write。调试时很有用："我想加点日志，但老是不小心去'修'一些不相干的代码。"

## 分发技能

技能最大的好处之一，就是你可以把它们分享给团队里的其他人。

把技能分享给别人，你可能会想用两种方式：

- 把你的技能签入仓库（放在 `./.claude/skills` 下）。
- 做一个插件，再搭一个 Claude Code 插件市场，让用户上传和安装插件（更多内容见[文档](https://code.claude.com/docs/en/plugins)）。

对于跨越相对较少几个仓库的小团队，把技能签入仓库这个方式很好用。但每一个签入的技能，也都会给模型的上下文增加那么一点点负担。当你规模做大时，一个内部插件市场让你能够分发技能、让团队自行决定装哪些，还能附带一套初始设置流程。

## 经营一个技能市场

怎么决定哪些技能进市场？人们又怎么提交它们？

在 Anthropic，我们没有一个集中的团队来拍板；相反，我们试着有机地发现那些最有用的技能。如果有人做了个技能、想让大家试试，他可以把它上传到 GitHub 上的一个 sandbox 文件夹，然后在 Slack 或其他场合把大家引过去。

一旦某个技能有了热度（这由技能的拥有者来判断），他们就可以提个 PR，把它挪进市场。

## 组合技能

你可能会想要彼此依赖的技能。比如，你可能有一个上传文件的文件上传技能，和一个生成 CSV 并上传它的 CSV 生成技能。这种依赖管理目前还没有原生内建到市场或技能里，但你只要按名字引用其他技能，模型就会在它们已安装的情况下调用它们。

## 度量技能

为了搞清楚一个技能表现如何，我们用了一个 PreToolUse hook，让我们能在公司内部记录技能的使用情况（示例代码在此）。这意味着我们能找出那些受欢迎的技能，或者那些触发频率低于我们预期的技能。

## 上手开始

技能的最佳实践仍在不断演进。我们最好的那些技能，大多一开始只是寥寥几行加上一个坑点，后来之所以变好，是因为人们随着 Claude 不断撞上新的边缘情况，持续往里添东西。

理解技能最好的方式，就是上手开始、动手试验，看看什么对你管用。
