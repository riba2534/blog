# 在 Codex 中使用 Goals：让长任务持续推进的持久目标

Goal 是 Codex 里的**持久目标**：它让一个会话（thread）跨越多轮，持续朝着一个明确的结果推进。一个 Goal 给 Codex 设定的是一个**完成条件**——什么应该为真、成功如何被验证、哪些约束必须始终保持不被破坏。

Codex 处理边界清晰的编码任务本就很顺手：检视一个仓库、修一个 bug、加一个测试、解释一个失败、实现一处聚焦的改动。Goal 面向的是另一类任务——下一步该做什么，取决于 Codex 一路上学到了什么：性能剖析、打补丁、跑基准、复现一个偶发失败的测试，或者把一个研究问题变成有证据背书的审计。

这类任务**并不需要一个更大的 prompt，它们需要的是一个持久的目标**。有了 Goal，Codex 就能始终盯住这个目标，自己判断工作是否已经完成，并选择下一个有用的动作，而不需要你在每一个中间结果之后都把目标重述一遍。

Goal 不是没有边界的后台自主运行。**它是一份有范围、由用户控制的"完成契约"**。你来定义结果，Codex 对照会话里的证据干活，而这个 Goal 可以被暂停、恢复、清除、完成，或者因预算用尽而停止。

## 你将学到什么

读完这篇指南，你将能够：

- 判断什么时候用 Goal 比用一次性 prompt 更合适。
- 写出一个有可度量结果、有验证面、有约束的 Goal。
- 用 `/goal`、`/goal pause`、`/goal resume`、`/goal clear` 管理 Goal 的整个生命周期。

## 前置条件

- 一个支持 Goal 的 Codex 版本。
- 一个有明确终点线的任务，并且有一个你能检视的证据来源——比如测试、一个基准（benchmark），或者一个最终产物。
- 足够的仓库或研究上下文，让 Codex 能去**验证**进展，而不只是**叙述**进展。

## 大纲

1. 启动一个 Goal 并管理它的生命周期。
2. 理解 Goal 与一次性 prompt 的区别。
3. 写出带有可审计完成标准的 Goal。
4. 把这套模式应用到复杂的研究工作上。
5. 判断什么时候普通 prompt 仍然是更好的工具。

## 快速上手：使用 Goals

当一个任务**有明确的终点线，但通往终点的路径不确定**时，就用 Goal。

适合的场景包括：性能优化、偶发失败测试的排查、依赖迁移、需要先复现的 bug 排查、多步重构、由基准驱动的调优，以及需要最终产物的研究任务。而对于一次性的小改动，普通 prompt 仍然是对的工具。

要使用 Goals，先安装或更新 Codex 并确认版本。**Goals 从 Codex 0.128.0 起可用。**

用 npm：

```bash
npm install -g @openai/codex@latest
codex --version
```

用 Homebrew：

```bash
brew update
brew upgrade --cask codex
codex --version
```

然后用 `/goal` 加上你想要的结果来设定一个 Goal：

```text
/goal 把 p95 延迟降到 120 ms 以下，且不让正确性测试出现回退
```

你可以在同一组命令里管理整个生命周期：

```text
/goal          查看当前的 Goal
/goal pause    暂停一个激活中的 Goal
/goal resume   恢复一个被暂停的 Goal
/goal clear    移除当前的 Goal
```

Goal 一旦激活，Codex 就能检视代码、运行相关命令、做改动、测试结果，并持续推进，直到抵达一个停止条件。这个停止条件可能是：成功、暂停、清除、被打断、预算上限，或者一个需要用户输入才能解开的阻塞点。

当你发现自己每一轮之后都在重复说下面这些话时，就该用 Goal 了：

```text
继续。
试下一个最可能的修复。
再跑一遍基准。
现在检查一下测试。
一直做到这事真的完成为止。
```

Goal 把这个意图**显式地表达了出来**。

## Goal 与 prompt 的区别

一个普通 prompt 说的是：**做接下来这件事**。

一个 Goal 说的是：**一直干，直到这个结果为真**。

这个区别很关键。在一次普通请求里，Codex 处理完眼前的指令，报告一个结果，然后等待。而有了 Goal，Codex 就有了一个绑定在会话上的、持久的目标。一轮结束之后，它能检视当前的证据，自己判断目标是否已经满足。如果答案是否，而 Goal 仍然激活、预算也还在，Codex 就能从最新的状态继续往下推进。

这让 Goal 在"正确的下一步取决于 Codex 刚刚学到了什么"的场景里最有用。例如：

```text
/goal 在 checkout 基准上，把 p95 下单延迟降到 120 ms 以下，同时保持正确性测试套件全绿
```

这不只是一句"提升性能"的请求。它给了 Codex 一个**可度量的结果**、一个**验证面**、一个**约束**。Codex 可以跑基准、检视热点路径、做一处有针对性的改动、重跑基准、跑正确性套件，如果结果还不够好就继续。

实际可用的心智模型很简单：

> **Prompt：** 提问 → 执行 → 结果 → 等待
>
> **Goal：** 执行 → 检查 → 继续或完成

Goal 给了 Codex 一条终点线。但活儿仍然必须**对照证据来审计**。

## 怎么写一个 Goal

一个好的 Goal 不只是一个更大的 prompt。它是一份紧凑的契约，规定了 Codex 应该怎么干、什么算成功，以及在成功还够不着的时候应该发生什么。

最强的 Goal 通常会定义六件事：

- **结果（Outcome）**：工作完成时，什么应该为真。
- **验证面（Verification surface）**：那个能证明结果的测试、基准、报告、产物、命令输出或源材料。
- **约束（Constraints）**：在 Codex 干活的过程中，什么绝不能回退。
- **边界（Boundaries）**：Codex 可以动用哪些文件、工具、数据、仓库或资源。
- **迭代策略（Iteration policy）**：每次尝试之后，Codex 该怎么决定下一步试什么。
- **阻塞停止条件（Blocked stop condition）**：在当前限制下已经没有站得住脚的路径时，Codex 应该在什么时候停下来并报告。

一个好用的模板是：

```text
/goal <期望的最终状态>，由 <具体证据> 验证，同时保持 <约束>。只使用 <允许的输入、工具或边界>。每次迭代之间，<Codex 应如何选择下一个最优动作>。如果被阻塞或没有有效路径了，<Codex 应该报告什么，以及什么东西能解开进展>。
```

举个例子，下面这个 Goal 能用，但仍然偏单薄：

```text
/goal 把 p95 下单延迟降到 120 ms 以下，不让正确性测试回退
```

一个更强的版本，给了 Codex 一份更完整的操作契约：

```text
/goal 把 p95 下单延迟降到 120 ms 以下，由 checkout 基准验证，同时保持正确性套件全绿。只动用 checkout 服务、基准 fixture 以及相关测试。每次迭代之间，记录改了什么、基准显示了什么、以及下一个最值得做的实验。如果基准跑不起来或没有有效路径了，就停下，并给出已尝试的路径、已收集的证据、阻塞点，以及下一步需要什么输入。
```

对于研究和调查类任务，原则是一样的。**在动手之前先定义好证据标准**，尤其是在精确的证明可能拿不到的时候：

```text
/goal 用现有的论文材料和本地资源，对这篇论文做出证据最充分的复现。在可行的地方尝试复现核心结果，在可能的地方验证输出，最后给出一份报告，把"已确认的发现""近似的重建""被阻塞的论断""仍存在的不确定性"分开陈述。
```

这种 Goal 给了 Codex 调查的空间，同时让最终结果保持诚实。它说的不只是"继续干"，**它定义了"完成""被阻塞""仍不确定"到底各自意味着什么**。

当任务很清楚、但 Goal 还没想好的时候，Codex 可以帮你把 Goal 写出来。一个简单的两步工作流很好用：第一步，用大白话描述这项工作，让 Codex 把它转成一个 Goal 草稿；第二步，审阅这份草稿，在激活之前把成功条件、验证面、约束和阻塞停止条件收紧。

例如：

```text
帮我把这个变成一个强的 /goal：我想让 Codex 持续在这个偶发失败的 checkout 测试上推进，直到我们要么有证据地修好它，要么能清楚地解释清楚是什么在阻塞进展。
```

Codex 随后就能提出一个更完整的 Goal，问出真正必要的缺失细节，并给你留下一个更干净的 `/goal` 可以直接用。

## Goal 激活之后，有什么变了

当一个 Goal 激活时，有三件事变了。

**第一，目标始终可见。** 如果 Codex 跑了一个测试、它失败了，会话里仍然保留着最初的目标。如果一个基准有改善但没达到阈值，Codex 可以继续。如果一条研究路径撞上了缺失的数据，Codex 可以调整证据计划，而不丢失对研究标准的视野。

**第二，从空闲会话继续推进成为可能。** Codex 不会在另一轮还在进行时、在有用户输入排队时、或在会话里还有其他活儿待办时继续推进。它只在会话空闲、Goal 激活且预算未尽时才继续。

**第三，完成必须基于证据。** 一个 Goal 不应该因为模型"觉得大概做完了"就被标记为完成。它应该只在目标对照过相关文件、测试、日志、基准输出、生成产物或其他具体证据之后，才算完成。

这就是设计的核心：**Codex 可以一直往前走，但由证据来决定它是否已经完成**。

## Codex 里的 Goal 是怎么设计的

Goal 被实现为**持久化的会话状态**，而不是全局记忆，也不是项目级的指令。这个设计选择很重要：目标属于那个承载着相关上下文的会话——包括 Codex 检视过的文件、它运行过的命令、它产出的 diff、它看到的日志，以及它一路建立起来的推理链路。

![一个 Goal 给会话增加了什么](https://img.riba2534.cn/images/2026/05/p1.png)

*图 2：Goal 给当前会话增加了持久状态、继续推进能力、控制开关以及证据校验。*

在架构层面，一个 Goal 是一份**持久的、会话作用域的状态**。它记录了 Codex 评估这个会话所需要的目标、生命周期、预算和进度账目。关键的边界在于"作用域"：Goal 属于当前会话，而不属于全局记忆或项目指令。

Codex 把这份状态当作用户、模型和会话三方之间的一份契约。一个 Goal 可以处于激活、暂停、完成或预算受限状态。这些状态决定了 Codex 是否可以继续、是否应该等待用户，以及是否应该总结进度而不是开始新的工作。

继续推进是**事件驱动的**，而不是一个简单的循环。Codex 只在安全的边界上检查是否要继续：一轮已经结束之后、没有其他工作待办时、没有用户输入排队时、并且会话空闲时。

调度器的行为是刻意保守的。**只做计划的工作不会触发继续推进。** 打断会暂停目标。恢复一个会话时，在合适的情况下可以恢复目标。如果某一轮继续推进没有产生任何工具调用，那么下一次自动继续就会被抑制，以免 Codex 空转。

![Codex 何时继续推进](https://img.riba2534.cn/images/2026/05/p2.png)

*图 3：只有当 Goal 激活、会话空闲、且没有用户输入排队时，Codex 才会继续推进。*

提示词层（prompting layer）强化了同样的架构。继续推进的提示词让 Codex 围绕激活的目标展开，但它们同时也要求在完成之前做一次审计。Codex 必须把目标和具体证据做对照：改过的文件、跑过的命令、通过的测试、基准输出、生成的产物，或者研究证据。

预算处理是显式的。当预算用尽时，Codex 应该停止实质性工作，总结进度和阻塞点，并指出下一个有用的步骤。**触及预算上限，并不等于完成了目标。**

工具契约把生命周期的权限框定得很有限。模型可以启动一个 Goal，也只能在证据支持"完成"时，才把一个已存在的 Goal 标记为完成。暂停、恢复、清除，以及预算受限的状态转换，仍然由用户或系统控制。

这就是要记在心里的架构：**一个 Goal 是一份会话作用域的完成契约。** 它把持久的目标状态、生命周期控制、继续推进策略、预算账目和基于证据的完成揉到了一起。重点不是让 Codex 永远循环下去，而是让目标一直存续，直到证据说工作已经完成。

## 把一个弱 Goal 变成一个强 Goal

弱：

```text
/goal 提升性能
```

强：

```text
/goal 在 checkout 基准上把 p95 延迟降到 120 ms 以下，同时保持正确性测试套件全绿
```

### 例子：性能调优

![把 Goal 写成完成条件](https://img.riba2534.cn/images/2026/05/p3.png)

*图 4：强 Goal 会点名最终状态、验证面和约束。*

更强的那个版本给了 Codex 三样东西：一个**结果**、一个**验证方法**、一个**约束**。它还给了 Codex 一种"知道什么时候不该停"的依据。如果 p95 从 180 ms 改善到 135 ms，Goal 没完成。如果延迟降到 120 ms 以下、但正确性测试挂了，Goal 没完成。如果基准跑不起来，Codex 必须把这个阻塞点暴露出来，而不是宣布成功。

同样的规则在性能工作之外也成立：**Goal 应该具体到足以验证，又开放到足以支撑探索**。

一个 Goal 应该窄到足以审计，又宽到足以让 Codex 自己选下一个动作。"修好这个失败的 checkout 测试"可能太窄了——如果真正的问题出在上游的一个依赖上。"改进整个系统"又太宽了——因为没有可审计的面。"在当前分支上让 checkout 测试套件通过，且不改变公开 API 的行为"就好得多。

同样的原则适用于生成类产物。一个弱 Goal 说：

```text
/goal 给这个特性写文档
```

一个更强的 Goal 说：

```text
/goal 为 Goals 产出一个文档页面，讲清楚生命周期、命令面，并给两个例子。验证这个页面能在本地构建成功，且所有引用到的命令都和当前 CLI 的行为一致。
```

第二个 Goal 给了 Codex 一些它可以**检视**的东西：一个页面、一个构建命令、命令的行为。

这套纪律在**研究类 Goal** 上更为重要。在调查开始之前就定义好证据标准：什么算精确复现、什么算部分重建、什么算代理性的旁证支持、什么应该被当作被阻塞。

一个强的研究类 Goal，应该要求 Codex 建立一份**论断清单**，把论断映射到证据上，把可行的部分实现出来，给阻塞点打上标签，并产出一份审计——把"已确认的论断""仅有旁证支持的""被阻塞的论断"和"仍存在的不确定性"分开。

这让 Goal 窄到足以审计，又不至于把整条路径都规定死。Codex 可以自己选下一个动作，但完成标准是固定的。

## 用 Goal 做复杂研究：复现一篇量化论文

下面是一个用上这些原则的研究类 Goal 的具体例子。

这个案例研究的对象是 Buehler、Gonon、Teichmann 和 Wood 的论文《Deep Hedging》。这篇论文要问的是：在不同的风险偏好、交易成本和更高维的市场设定下，神经网络交易策略能否复现基于模型的对冲。正确的 Goal 不是抽象地"复现这篇论文"，而是**去尝试论文里的核心数值论断，把精确的机制与近似的训练替代物区分开，并且对"哪些东西无法从现有材料里精确重放"保持明确**。

一个弱的研究类 Goal 会是：

```text
/goal 复现 Buehler 等人的《Deep Hedging》
```

这太欠规定了。它没说哪一节是关键、什么算复现、训练状态拿不到时怎么办，也没说怎么把"数值上很接近"和"精确重放"区分开。

一个更好的 Goal 是：

```text
/goal 用现有的论文材料和本地资源，对 Buehler 等人的《Deep Hedging》做出证据最充分的复现。尝试复现每一个核心结果，验证输出，最后给出一份报告，把"已复现的机制""近似的训练结果""被阻塞的精确重放"和"仍存在的不确定性"分开陈述。
```

更强的这个版本之所以管用，是因为它**点名了证据标准和最终产物**。Codex 不只是想搞出一个看起来很唬人的复现，而是要在不夸大现有证据所能支撑的范围的前提下，把不确定性降到最低。

![研究类 Goal 把论文拆成证据通道](https://img.riba2534.cn/images/2026/05/p4.png)

*图 5：研究类 Goal 会先把论文分解成一条条证据通道，然后再宣布状态。*

实践中，这个 Goal 给整个调查提供了一份具体的操作契约。

Codex 用它来：

- 把核心论断和支撑性论断分开，
- 把这些论断映射到现有的证据上，
- 把那些可以在本地测试的部分重建出来，
- 并给那些无法从现有材料精确复现的论断打上标签。

有几块是可行的。Codex 重建了定价和对冲的机制，复现了 Heston 参考价格，为 CVaR 对冲实验训练了策略，重建了主要的直方图和对冲曲面产物，复现了 Black-Scholes 交易成本斜率，并为 Heston 交易成本和高维例子跑了训练后的校验。

有些论断仍然被缺失的源材料阻塞了。这篇论文没有提供确切的随机种子、生成的训练路径、TensorFlow 计算图、优化器状态、检查点（checkpoint）或完整的原始模拟状态。**这意味着最强的、诚实的结果是一个部分且近似的复现，而不是一次精确的神经网络重放。**

这正是 Goal 的价值所在。它让工作在阻塞点出现之后仍然往前走，但同时也让最终的措辞保持诚实。一个训练出来的替代物可以**支持**一个论断，一个数值上的接近匹配可以**提升信心**，一张重建出来的图可以**验证**结果的一部分——但这些都不应该被描述成"精确地恢复了原始实验"。

![最终输出应保留不同的置信层级](https://img.riba2534.cn/images/2026/05/p5.png)

*图 6：最终的输出应该保留不同层级的认知支持强度。*

最终的报告应该保留这些不同层级的支持强度，而不是把它们压平成一句单一的"成功"论断。

举个例子，一条账目记录可以长这样：

```text
论断：在无交易成本的情况下，深度对冲近似了完整市场下的 Heston 对冲。
路径：重建了模型机制、参考对冲对比、并训练了神经网络策略。
证据面：价格校验、直方图、对冲曲面。
状态：接近的近似复现。
仍存的不确定性：原始训练路径、种子和检查点不可得。
```

这就是 Goal 在研究中的演示价值。它让 Codex 能在模糊地带里持续推进，同时防止一个"看似合理的产物"变成一个"被夸大的结论"。Goal 要求 Codex 做的不只是"完成"，**它定义了"完成"是什么意思**：一份逐条论断、扎根于证据的审计，对近似明确标注，并对"复现"和"重放"之间的边界保持诚实。

![Codex 中输入研究类 Goal 的界面](https://img.riba2534.cn/images/2026/05/p6.png)

## 什么时候不该用 Goal

Goal 并不是所有任务都适用的工具。

**不要**给一行改动、一个简单解释、一次简短代码审查，或者一个"你想要一个答案然后就停"的问题用 Goal。这些场景用普通的 Codex prompt 更好。

**不要**在终点线模糊时用 Goal。"把这个弄好点"没有给 Codex 任何可靠的完成条件。"重构这段代码"也很弱——除非你定义了预期的最终状态、测试和约束。

**不要**用 Goal 来掩盖不确定性。如果数据可能拿不到，就在 Goal 里说清楚。如果一个基准可能不稳定，就说怎么处理。如果允许用代理性的旁证，就定义它应该怎么被标注。

Goal 最强的时候，是任务同时具备这三个属性：**一个持久的目标、一条基于证据的终点线、以及一条可能需要好几轮调查才能走完的路径**。

## 结语：让目标存续，但让证据决定

Goal 改变了 Codex 的运作模型。它把一个会话从"一连串孤立的 prompt"，变成了一个围绕明确结果的、有状态的工作循环。

这套架构是刻意有边界的。Goal 的作用域被框在一个会话里，携带着生命周期状态和预算账目，可以被暂停、恢复、清除、完成，或因预算用尽而停止。**Codex 可以一直往前走，但只能在用户定义的那份契约之内走。**

这让 Goal 在 Codex 本就最有价值的那些工作上特别有用：调试、优化、迁移、测试和研究。用户提供目标，Codex 跟着证据走。Goal 让两者始终连在一起，直到工作要么完成、要么被诚实地阻塞。

对于复杂的研究来说，这就是"生成一个答案"和"产出一份审计"之间的区别。**一个好的 Goal 要求 Codex 做的不只是"完成"，它告诉 Codex"完成"是什么意思。**
